## 마크다운 작성 규칙

1. 모든 마크다운 문서는 H1(`#`) 제목으로 시작한다.
2. 제목 아래에는 반드시 메타데이터 블록을 작성한다.
3. 섹션 구분은 H2(`##`)를 사용하고, 하위 항목은 H3(`###`)까지만 허용한다.
4. 표(table)는 마크다운 GFM 형식으로 작성한다. 헤더와 데이터 행 사이에 구분선(`---`)을 반드시 포함한다.
5. 코드 블록은 언어 식별자를 명시한다. SQL이면 ` ```sql `, Java이면 ` ```java `.
6. 링크는 상대 경로로 작성한다. 절대 URL은 외부 링크에만 사용한다.
7. 이미지 삽입은 `assets/` 경로를 참조한다.
8. 미확정 항목은 **미정** 으로 표기한다. 빈 칸으로 두지 않는다.

---

## 메타데이터 블록 형식

모든 문서는 제목 바로 아래에 다음 블록을 포함한다.

```markdown
- 상태: 작성 예정 / 초안 / 검토 중 / 확정
- 작성자:
- 마지막 수정일: YYYY-MM-DD
- 관련 요구사항: REQ-XXX-000, REQ-XXX-001 (해당 없으면 "없음")
- 관련 문서: 링크 (해당 없으면 "없음")
```

## 디렉토리 규칙

```
docs/
├── 00_project/       — 프로젝트 전반 (개요, 기술 스택, 규칙)
├── 01_requirements/  — 요구사항 (기능, 비기능, 추적 매트릭스)
├── 02_domain/        — 도메인 설계 (개요, 시나리오, 유스케이스)
├── 03_ui/            — UI 설계 (화면 목록, 흐름, 와이어프레임)
├── 04_database/      — DB 설계 (데이터 소스, ERD, 테이블 명세, DDL)
├── 05_api/           — API 설계 (원칙, 명세)
├── 06_backend/       — 백엔드 설계 (아키텍처, 패키지, 클래스, 에러 처리)
├── 07_algorithm/     — 알고리즘 기획 (A* 경로 탐색)
├── 08_schedule/      — 일정 (WBS, 간트 차트)
├── 09_submission/    — 제출 (체크리스트, 최종 README 계획)
└── assets/           — 이미지 및 다이어그램 파일
    ├── diagrams/     — ERD, 클래스, 유스케이스 다이어그램 이미지
    ├── wireframes/   — 와이어프레임 이미지
    └── screenshots/  — 실행 화면 캡처
```

## 파일 명명 규칙

- 파일명과 디렉토리명은 영어 소문자만 사용한다.
- 단어 구분은 하이픈(`-`)으로 한다. 언더스코어나 카멜케이스는 사용하지 않는다.
- 마크다운 파일 확장자는 `.md`로 통일한다.
- SQL 파일 확장자는 `.sql`로 통일한다.
- 이미지 파일은 `{내용}-{날짜}.png` 형식을 권장한다. 예: `erd-20260514.png`

## 요구사항 ID 규칙

형식: `REQ-{도메인}-{세 자리 숫자}`

| 접두어 | 도메인 |
|--------|--------|
| REQ-HOUSE | 주택 거래 정보 |
| REQ-MEMBER | 회원 관리 |
| REQ-AUTH | 인증/인가 |
| REQ-FAVORITE | 관심 지역 |
| REQ-COMMERCIAL | 상권 정보 |
| REQ-ENV | 환경 정보 |
| REQ-ROUTE | 경로 탐색 |
| REQ-NOTICE | 공지사항 |

- 번호는 001부터 순차 부여한다.
- 동일 ID는 요구사항 문서, UI, API, DB, 백엔드, 알고리즘 문서에서 일관되게 사용한다.
- 삭제된 요구사항 ID는 재사용하지 않는다.

## 다이어그램 에셋 경로 규칙

| 유형 | 경로 |
|------|------|
| ERD 이미지 | `assets/diagrams/erd-YYYYMMDD.png` |
| 클래스 다이어그램 | `assets/diagrams/class-{도메인}-YYYYMMDD.png` |
| 유스케이스 다이어그램 | `assets/diagrams/usecase-YYYYMMDD.png` |
| 와이어프레임 | `assets/wireframes/{화면ID}-YYYYMMDD.png` |
| 실행 화면 캡처 | `assets/screenshots/{화면ID}-YYYYMMDD.png` |

마크다운에서 이미지를 삽입할 때는 항상 상대 경로를 사용한다.

```markdown
![ERD](../assets/diagrams/erd-20260514.png)
```

## 한국어 작성 규칙

- 본문은 한국어로 작성한다.
- 기술 용어(API, DB, HTTP, Spring Boot 등)는 영어를 유지한다.
- 표 헤더, 코드 블록, 파일명, ID는 영어를 허용한다.
- 번역투 표현(`~을 통해`, `~에 대해`, `~되어진다`)은 사용하지 않는다.
- 불필요한 이모지는 사용하지 않는다.
- 문장은 간결하게 유지한다. 한 문장에 하나의 사실만 담는다.

## 문서 간 링크 규칙

- 문서 내에서 다른 문서를 참조할 때는 항상 상대 경로 링크를 사용한다.
- 특정 섹션을 참조할 때는 앵커를 포함한다.

```markdown
[API 명세 참조](../05_api/api-spec.md#회원-api)
[요구사항 추적](../01_requirements/requirement-traceability.md)
```

- 외부 공공 데이터 링크는 URL 전체를 표기한다.
